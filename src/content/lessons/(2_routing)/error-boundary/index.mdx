Next.js provides a powerful error handling system through special `error.tsx` files.
These files create error boundaries that gracefully catch and display errors while keeping the rest of your application functional.

## What are Error Boundaries?

Error boundaries in Next.js are React components that:

- Catch JavaScript errors anywhere in their child component tree
- Log those errors
- Display a fallback UI instead of the crashed component tree

## How Error Boundaries Work

When you create an `error.tsx` file, Next.js automatically wraps the page or component in an error boundary. If an error occurs, the error UI is displayed while keeping the rest of the application interactive.

An example of what an `error.tsx` file might look like:

```tsx
// app/error.tsx
'use client'

type ErrorProps = {
  error: Error
  reset: () => void
}

export default function Error({ error, reset }: ErrorProps) {
  return (
    <div className="p-4 rounded-lg bg-red-50 border border-red-200">
      <h2 className="text-red-800 font-semibold">Something went wrong!</h2>
      <p className="text-red-600 mt-2">{error.message}</p>
      <button
        onClick={reset}
        className="mt-4 px-4 py-2 bg-red-100 text-red-700 rounded hover:bg-red-200">
        Reload
      </button>
    </div>
  )
}
```

These files can be used to handle errors in any part of the application, this includes things like,
API calls, database errors, and other unexpected errors.

To put it simply, `error.tsx` files will catch any errors that occur and display a fallback UI.

## Key Concepts

### 1. Client Components

- Error components must be Client Components (marked with `'use client'`)
- They receive `error` and `reset` props automatically
- The `reset` function allows users to retry the failed operation ( This essentially refreshes the page)

### 2. Error Recovery

You can use something like Sentry or bugsnag to log errors to a service via `error.tsx` files.

```tsx
'use client'

type ErrorProps = {
  error: Error
  reset: () => void
}

export default function Error({ error, reset }: ErrorProps) {
  useEffect(() => {
    // Log the error to an error reporting service
    logger.error(error)
  }, [error])

  return (
    <div className="p-4 rounded-lg bg-red-50 border border-red-200">
      <h2 className="text-red-800 font-semibold">Something went wrong!</h2>
      <p className="text-red-600 mt-2">{error.message}</p>
      <button
        onClick={reset}
        className="mt-4 px-4 py-2 bg-red-100 text-red-700 rounded hover:bg-red-200">
        Reload
      </button>
    </div>
  )
}
```

### 3. Nested Error Handling

Error boundaries can be nested, this means that if an error occurs in a child component, it will be caught by the nearest parent error boundary.

This means if you place a `error.tsx` at the root level of the app folder, and an error occurs in a child component,
the whole app will render the error UI.

So it is best practice to have multiple `error.tsx` files for different sections in your app, each can have their own
fallback UI so that if any part fails you can keep the whole app functional.

## Best Practices

1. **Graceful error handling**

   - Keep error messages user-friendly
   - Provide clear recovery actions
   - Maintain app functionality outside the error boundary

2. **Error Logging**

   ```tsx
   export default function Error({ error }: ErrorProps) {
     useEffect(() => {
       // Send to your preferred error tracking service
       reportError(error)
     }, [error])

     return <ErrorUI />
   }
   ```

## Common Patterns

### 1. Contextual Error UI

```tsx
'use client'

export default function ContextualError({ error }: ErrorProps) {
  if (error instanceof NetworkError) {
    return <NetworkErrorUI />
  }

  if (error instanceof ValidationError) {
    return <ValidationErrorUI />
  }

  return <GenericErrorUI />
}
```

## Working with Server Components

Error boundaries work seamlessly with Server Components:

```tsx
// page.tsx
export default async function Page() {
  const data = await fetchData() // If this throws, error.tsx handles it
  return <DataDisplay data={data} />
}
```
